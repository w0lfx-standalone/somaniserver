# v1.5 – Full Throttle: Automation, Secure Access, Kasm & Network Security

## Objective
The "Full Throttle" milestone represents the transition of `somaniserver` from a collection of manually managed services into a resilient, autonomous, and secure private cloud ecosystem. This phase focused on achieving **High Availability (HA) through automation** and **Zero-Trust networking**. The goal was to ensure the server could recover from power failures without human intervention, manage complex service dependencies, and provide a secure, filtered, and private gateway for all devices—local or remote.

---

## 1. Multi-Tier Automation & Resilience
To achieve a truly "set and forget" infrastructure, we implemented a cascading automation strategy. Each layer ensures the next is ready, creating a reliable chain of recovery.

### 1.1 Hardware Layer: BIOS Power Orchestration
True autonomy begins before the OS even loads. We modified the power management settings in the physical host's BIOS (Intel i3-6100 based system).
*   **The Setting:** Enabled **"Restore on AC Power Loss"** (set to "Power On").
*   **Technical Rationale:** In a home environment, power fluctuations or outages are inevitable. This hardware-level instruction ensures the physical machine attempts a boot sequence the moment current is restored to the PSU, removing the need for a manual power-button press.

### 1.2 Hypervisor Layer: Proxmox Dependency Management
Virtualization adds a layer of complexity; if the services VM (Ubuntu) boots before the storage VM (TrueNAS), NFS mounts will fail, and the Docker containers will crash. 
*   **Boot Order (Start/Shutdown):**
    1.  **TrueNAS SCALE (ID 100):** Priority 1. Start Delay: **0s**.
    2.  **Ubuntu Server (ID 101):** Priority 2. Start Delay: **60s**.
*   **Technical Rationale:** The 60-second delay is intentional. TrueNAS requires time to initialize the ZFS pool (`swimming_pool`), verify the pool's health, and start the NFS export daemon. The delay ensures that when the Ubuntu VM reaches its "mount" stage, the network target is already broadcasting and ready to accept the connection.

![Proxmox Boot Order Screen](https://github.com/w0lfx-standalone/somaniserver/main/assets/proxmox_boot_order.png)
*Caption: Orchestrating the VM startup sequence to respect storage dependencies.*

### 1.3 Service Layer: systemd Orchestration
While Docker restart policies (`restart: always`) are useful, they don't respect host-level dependencies like NFS availability. We integrated our Docker stacks directly into **systemd**.
*   **Implementation:** Created custom `.service` units in `/etc/systemd/system/` (e.g., `immich.service`, `seafile.service`, `kasm.service`).
*   **Technical Rationale:** By tying the Docker stacks to `network-online.target`, we ensure that the OS has a stable network stack before attempting to pull or run containers. This significantly reduces "race conditions" where Docker tries to start before the NFS shares are fully mounted into the filesystem.

## 2. Secure Remote Access via Tailscale
Standard remote access methods (Port Forwarding, Public DNS, DMZ) are security risks. For `somaniserver`, we adopted a **Zero-Exposure** approach using **Tailscale**.

*   **Technology:** Tailscale uses the **WireGuard** protocol to create a peer-to-peer (P2P) mesh VPN. 
*   **Deployment:** Installed on Proxmox (Host), TrueNAS (Storage), and Ubuntu (Services).
*   **The "Magic":** 
    *   **No Port Forwarding:** Tailscale uses NAT traversal to establish connections, meaning my home router's firewall remains completely closed to the outside world.
    *   **Encrypted Mesh:** Every device on the "Tailnet" is authenticated and every packet is end-to-end encrypted. I can access my photos on Immich or manage files on Seafile from an airport WiFi as securely as if I were at home.

## 3. Kasm Workspaces: Containerized Isolation
To provide a secure way to access the web from within the server environment, we deployed **Kasm Workspaces**. This isn't just a browser; it's a streaming, disposable desktop environment.

*   **Deployment Strategy:** Standard Docker-based installation for maximum flexibility.
*   **Storage Optimization:** 
    *   **Localized Speed:** Kasm images (the OS layers) are stored on the Ubuntu VM's local HDD for rapid startup.
    *   **Centralized Persistence:** All user profiles, downloads, and workspace configurations are mapped via NFS to the redundant **ZFS Mirror Pool** (`/mnt/swimming_pool/kasm`).
*   **Security Impact:** If a session is compromised or simply finished, it is "burned." A fresh, clean container is spawned for the next session, ensuring that untrusted browsing never leaves a trace on the host system.

## 4. Network-Wide Security & Privacy: AdGuard Home + Unbound
The final pillar of "Full Throttle" is sovereign network security, handled by a **Raspberry Pi Zero 2 W**. This node acts as the "Brain" of the network's DNS.

### 4.1 DNS Filtering with AdGuard Home
We replaced the default router DNS with **AdGuard Home**.
*   **Impact:** Every device in the house (Smart TVs, Phones, Laptops) is automatically protected. Ad-trackers, malware domains, and telemetry are blocked at the DNS level before they ever reach a device.

### 4.2 Sovereignty with Unbound
Instead of forwarding DNS queries to Google (8.8.8.8) or Cloudflare (1.1.1.1), we integrated **Unbound** as a recursive DNS resolver.
*   **Technical Rationale:** Unbound communicates directly with the **Root DNS Servers**. This eliminates the "middleman," preventing upstream providers from logging my browsing habits. It ensures that my DNS resolution is as private and independent as possible.

### 4.3 Remote Protection (Global DNS)
By integrating the Pi's Tailscale IP into the Tailnet's **Global Nameserver** settings:
*   **Result:** When I leave the house and my phone switches to LTE, Tailscale automatically routes my DNS queries back to the Raspberry Pi. I get the same ad-blocking and Unbound privacy protection anywhere in the world.

## 5. Verification & Reliability Testing
To confirm the success of this milestone, we conducted a **Cold Start Stress Test**:
1.  **Simulated Failure:** Pulled the main power cable from the server.
2.  **Autonomous Recovery:** Reconnected power and timed the recovery.
3.  **Metrics:**
    *   **Physical Host Boot:** ~45 seconds.
    *   **TrueNAS Readiness:** ~20 seconds.
    *   **Ubuntu Service Startup:** ~60 seconds (post-delay).
    *   **Total Uptime:** Full service availability (Immich/Seafile/Kasm) reached within **under 3 minutes** with zero manual commands issued.
4.  **Security Audit:** Verified that no ports were open on the router and that DNS queries from remote devices were being filtered by AdGuard Home.
